# Plan: v0.2.0 — MVP (first full season)

## Context

0.2.0 completes the product for a single user running ~8 scans across one season. The capture-to-analysis loop works (0.1.2). This milestone adds the inventory scan, visualization, export, and operational polish needed for real field use.

The app already has: vineyard CRUD, scan CRUD, video capture with GPS, Gemini integration (BBCH per vine), Git sync, CI/CD to rebbergverwaltung.haeusler-wein.ch.

---

## What gets built

### 1. Inventory scan (VineMap build)

The first scan of the season establishes which vines exist in each row. This is the baseline for all later scans.

**Gemini prompt update** (`app/src/lib/ai/gemini.ts`):
- Add `inventory` mode: when processing an inventory scan, the prompt asks Gemini to identify each vine stock and report its status (`present`, `missing`, `dead`) in addition to BBCH
- Response format adds `status` field: `[{ vine_index, bbch_pred, confidence, status }]`
- `parseGeminiResponse` builds both `BbchResult[]` and `VineMap[]` from inventory responses

**Scan type** — add `is_inventory` boolean to `Scan` interface:
- When creating a scan, user can toggle "Inventory scan" (checkbox)
- Inventory scans update/replace the VineMap for the vineyard
- Regular scans use existing VineMap vine count as context for Gemini

**Processor update** (`app/src/lib/ai/processor.ts`):
- When processing an inventory scan video, also store VineMap entries
- Clears previous VineMap for the vineyard before storing new entries (full rebuild)

**UI — new scan screen** (`app/src/routes/vineyard/[id]/scan/new/+page.svelte`):
- Add "Inventory scan" checkbox (default unchecked)
- Helper text: "First scan of the season. Records which vines are present, missing, or dead."

### 2. VineMap view

New route: `/vineyard/[id]/vinemap/+page.svelte`

Shows the inventory of vines per row:
- One section per row
- Each vine shown as a small badge: green (present), gray (missing), red (dead)
- Summary: total vines, present count, missing count, dead count
- Link from home page when VineMap exists

### 3. Scan results heatmap

New route: `/vineyard/[id]/scan/[scanId]/heatmap/+page.svelte`

Grid visualization: rows on Y-axis, vine indices on X-axis, cells colored by BBCH value.
- Color scale: cool (low BBCH) to warm (high BBCH)
- Tap a cell to see vine detail (vine index, BBCH, confidence)
- Link from scan detail page when results exist

### 4. Derived layers: ahead/behind, heterogeneity

Computed from BBCH results on the heatmap view:
- **Ahead/behind**: deviation from scan median — color scale from blue (behind) through white (median) to red (ahead)
- **Heterogeneity**: variance within each row — highlight rows with high internal spread
- Toggle between "BBCH", "Deviation", "Heterogeneity" views on the heatmap

**New module** — `app/src/lib/analysis/derived.ts`:
- `computeDeviation(results: BbchResult[]): { vine_index, row_number, deviation }[]`
- `computeRowHeterogeneity(results: BbchResult[]): { row_number, variance }[]`

### 5. Vine detail screen

New route: `/vineyard/[id]/vine/[row]/[vine]/+page.svelte`

Shows one vine across all scans:
- BBCH time series (simple line/dot chart using plain SVG or HTML)
- Table of scan date, BBCH, confidence, model version
- VineMap status (present/missing/dead) if inventory data exists
- Navigate from heatmap cell tap

### 6. Later-season tracking using VineMap

When a VineMap exists, pass the vine count per row to Gemini so it knows how many vines to expect. This is already partially implemented (processor checks VineMap), but needs:
- Consistent vine index alignment across scans (Gemini told to match VineMap indices)
- Prompt update to reference VineMap: "Row 3 has 37 present vines (vine 12 is missing, vine 28 is dead)"

### 7. Video purge after DONE

Free device storage by removing video blobs after successful processing.

**UI** — scan detail page:
- "Purge videos" button appears when all videos in a scan are DONE
- Deletes video blobs from IndexedDB, keeps metadata and results
- Shows freed space estimate

**Storage update** (`app/src/lib/storage/idb.ts`):
- `purgeVideoBlob(videoId: string)` — deletes blob, clears `local_uri` on RowVideo

### 8. CSV export

Export scan results and vine time series as CSV.

**New module** — `app/src/lib/export/csv.ts`:
- `exportScanCsv(scanId: string): string` — `scan_id, date, row_number, vine_index, bbch_pred, confidence`
- `exportVineTimeSeriesCsv(vineyardId: string): string` — `row_number, vine_index, scan_date, bbch_pred`
- Trigger download via blob URL

**UI** — home page:
- "Export" button (dropdown: "Scan CSV", "Time series CSV")
- Only visible when results exist

### 9. Retry / re-record for FAILED videos

Already partially implemented (retry button exists on scan detail). Needs:
- Re-record option: navigate to record screen with same row/direction pre-filled
- Replace the existing video blob and reset status to LOCAL_ONLY

### 10. Offline capture with deferred upload

Already works by design — videos are recorded offline and stored locally. Needs polish:
- Clearer offline indicator showing "X videos pending processing"
- Disable "Process" button when offline (check `navigator.onLine`)
- When coming back online, show notification that videos are ready to process
