# Plan: v0.1.1 — Video capture + upload queue

## Context

0.1.1 adds video recording and local queuing. After this milestone, a user can create a scan, record one video per row, and see queued videos with their status. No cloud upload yet — that's 0.1.2.

The app already has: vineyard CRUD (with configurable row count and walking direction), settings, PWA shell, IndexedDB persistence via `idb-keyval`.

---

## What was built

### 1. Scan CRUD

Scans are the container for a set of row videos. A scan belongs to a vineyard.

**Data model** — `Scan` in `types.ts`:
```
{ id, vineyard_id, created_at, note }
```

**Storage layer** (`app/src/lib/storage/idb.ts`):
- `saveScan(scan)`, `loadScan(id)`, `deleteScan(id)`, `listScans(vineyardId)`
- Key format: `scan:<id>`, filtered by `vineyard_id`
- `deleteScan` cascades to row videos and their blobs

**Store** (`app/src/lib/stores/scan.ts`):
- `scans` writable store (scans for current vineyard)
- `loadScans(vineyardId)`, `addScan(vineyardId, note)`, `removeScan(id)`

### 2. RowVideo CRUD + blob storage

Each row video has metadata (RowVideo entity) and a video blob.

**Data model** — `RowVideo` in `types.ts`:
```
{ id, scan_id, row_number, direction, local_uri, cloud_uri, status, gps_start?, gps_end?, created_at }
```

- `direction`: `top_to_bottom | bottom_to_top` (display labels adapt based on vineyard's `direction_label`)
- `gps_start`, `gps_end`: optional `GpsPoint` (`{ lat, lng }`) captured during recording
- `local_uri` is the IndexedDB key for the video blob: `video-blob:<id>`

**Storage layer** (`app/src/lib/storage/idb.ts`):
- `saveRowVideo(rowVideo)`, `loadRowVideo(id)`, `deleteRowVideo(id)`, `listRowVideos(scanId)`
- Key format: `rowvideo:<id>`, filtered by `scan_id`
- `saveVideoBlob(id, blob)`, `loadVideoBlob(id)`, `deleteVideoBlob(id)`
- Key format: `video-blob:<id>`
- `deleteRowVideo` also deletes the associated blob

**Store** (`app/src/lib/stores/scan.ts`):
- `rowVideos` writable store (videos for current scan)
- `loadRowVideos(scanId)`, `addRowVideo(scanId, rowNumber, direction, blob, gpsStart?, gpsEnd?)`, `removeRowVideo(id)`

### 3. Video capture screen

Route: `/vineyard/[id]/scan/[scanId]/record/+page.svelte`

**UI flow:**
1. User selects row number (1–N dropdown, N = vineyard's `row_count`) and direction (labels adapt to vineyard's `direction_label`: Top/Bottom or Left/Right)
2. Taps "Start recording" — camera via `getUserMedia` (rear camera), starts `MediaRecorder`
3. GPS tracking (`watchPosition`) runs during recording — captures start/end position
4. Taps "Stop" — stops recording, assembles blob from chunks, stops camera
5. Preview shows recorded clip with playback controls
6. "Save" persists blob + RowVideo to IndexedDB with `status: LOCAL_ONLY`, navigates back to scan detail
7. "Discard" releases blob, re-opens camera for another take

**Technical notes:**
- `navigator.mediaDevices.getUserMedia({ video: { facingMode: 'environment' } })` for rear camera
- `MediaRecorder` with `video/mp4` (fallback to `video/webm` if unsupported)
- Collect chunks via `ondataavailable`, assemble `Blob` on stop

### 4. Scan detail screen

Route: `/vineyard/[id]/scan/[scanId]/+page.svelte`

Shows:
- Scan note and date
- List of recorded row videos (row number, direction icon from vineyard config, status badge)
- "Record row" button → navigates to record screen
- "Delete scan" button with confirmation

### 5. New scan screen

Route: `/vineyard/[id]/scan/new/+page.svelte`

- Note field (optional, e.g. "after rain")
- "Create" → creates Scan, navigates to scan detail

### 6. Home page update

Updated `+page.svelte`:
- Scan list for selected vineyard (date, note, video count) — directly on home
- Each scan links to its detail page
- "New scan" button when a vineyard is selected
- Upload queue status displayed per video as `LOCAL_ONLY` badge (no vineyard detail page — scans are managed from home)

## What is NOT in 0.1.1

- Actual upload to any cloud service (0.1.2)
- Gemini API calls (0.1.2)
- Upload retry or Wi-Fi-only toggle (0.2.0)
