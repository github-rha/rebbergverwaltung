# Plan: v0.1.2 — Gemini integration + Git sync + CI/CD

## Context

0.1.2 closes the capture-to-analysis loop. After this milestone, a user can record a row video, send it to Gemini for BBCH analysis, see results per vine, sync results to GitHub, and the app deploys automatically via CI/CD.

The app already has: vineyard CRUD, scan CRUD, video capture with GPS, local blob storage, IndexedDB persistence, PWA shell.

The app will be deployed to **rebbergverwaltung.haeusler-wein.ch** (custom domain on GitHub Pages).

---

## What gets built

### 1. Gemini API integration

Send a row video to Gemini and receive per-vine BBCH results.

**New module** — `app/src/lib/ai/gemini.ts`:
- `analyzeRowVideo(videoBlob: Blob, rowNumber: number, vineCount?: number): Promise<BbchResult[]>`
- Upload video via Gemini File API (for videos > inline limit)
- Send analysis prompt requesting per-vine BBCH classification
- Parse structured JSON response into `BbchResult[]` entities
- Record `model_version` from the API response

**Prompt design:**
- System prompt describes the task: identify individual vines in a row video, classify BBCH stage per vine
- Include row metadata (row number, expected vine count if known from VineMap)
- Request structured JSON output: `[{ vine_index, bbch_pred, confidence }]`
- Handle edge cases: missing vines, low visibility, partial row

**API details:**
- Use `generativelanguage.googleapis.com` REST API (direct fetch, no SDK dependency)
- API key from Settings (stored in IndexedDB)
- Model: `gemini-3.0-flash` (native video understanding, cost-effective)
- Video upload: use File API for large blobs, then reference `file_uri` in the generate request

### 2. Video processing state machine

Drive videos through `LOCAL_ONLY -> UPLOADING -> PROCESSING -> DONE / FAILED`.

**New module** — `app/src/lib/ai/processor.ts`:
- `processVideo(videoId: string): Promise<void>` — orchestrates the full pipeline for one video
- State transitions:
  1. Load video blob from IndexedDB
  2. Set status `UPLOADING` — upload to Gemini File API
  3. Set status `PROCESSING` — call generate endpoint with file reference
  4. Parse results -> store `BbchResult[]` -> set status `DONE`
  5. On any error -> set status `FAILED` with error message
- Each state transition persists to IndexedDB immediately (crash recovery)

**Store updates** (`app/src/lib/stores/scan.ts`):
- `updateVideoStatus(videoId: string, status: VideoStatus): Promise<void>`
- `processRowVideo(videoId: string): Promise<void>` — calls processor, refreshes store

**UI integration:**
- "Process" button on scan detail page (per video or "Process all")
- Status badges update in real-time as processing progresses
- Failed videos show error message and "Retry" option

### 3. BbchResult storage

**Storage layer** (`app/src/lib/storage/idb.ts`):
- `saveBbchResult(result: BbchResult): Promise<void>`
- `loadBbchResults(scanId: string): Promise<BbchResult[]>`
- `deleteBbchResults(scanId: string): Promise<void>`
- Key format: `bbch:<id>`, filtered by `scan_id`
- Cascade: deleting a scan also deletes its BbchResults

**Store** (`app/src/lib/stores/scan.ts`):
- `bbchResults` writable store
- `loadBbchResults(scanId: string): Promise<void>`

### 4. VineMap build from early-season scan

The VineMap records which vines exist in each row and their approximate positions.

**Storage layer** (`app/src/lib/storage/idb.ts`):
- `saveVineMapEntries(entries: VineMap[]): Promise<void>`
- `loadVineMap(vineyardId: string): Promise<VineMap[]>`
- `deleteVineMap(vineyardId: string): Promise<void>`
- Key format: `vinemap:<id>`, filtered by `vineyard_id`

**Building the VineMap:**
- When Gemini processes an early-season video, the response includes vine count and indices
- Store as VineMap entries: one per vine per row
- Later scans can reference the VineMap to know expected vine count per row
- VineMap is per-vineyard (not per-scan) — built once, updated if needed

**UI:**
- Show VineMap status on home page: "VineMap: 10 rows, 37 vines/row" or "No VineMap yet"
- Option to rebuild VineMap from a scan's results

### 5. Git-based result storage

Sync structured results to a GitHub repository.

**New module** — `app/src/lib/sync/github.ts`:
- `pushResults(vineyardId: string): Promise<void>` — push local results to GitHub
- `pullResults(vineyardId: string): Promise<void>` — pull results from GitHub to local
- Uses GitHub Contents API (`PUT /repos/:owner/:repo/contents/:path`)
- Authentication: PAT from Settings
- File format: `data/<vineyard-name>/scans.json` (all scans + results for a vineyard)
- Conflict handling: last-write-wins (single user, manual sync)

**Data format** (committed to repo):
```json
{
  "vineyard": { "id": "...", "name": "...", "row_count": 10, "direction_label": "top_bottom" },
  "vine_map": [ ... ],
  "scans": [
    {
      "id": "...", "created_at": "...", "note": "...",
      "results": [
        { "row_number": 1, "vine_index": 1, "bbch_pred": 55, "confidence": 0.92, "model_version": "gemini-3.0-flash" },
        ...
      ]
    }
  ]
}
```

**UI integration:**
- "Sync" button on home page (visible when GitHub settings are configured)
- Sync status indicator (last synced timestamp)
- Manual push/pull only (no auto-sync)

### 6. CI/CD pipeline

GitHub Actions workflow to build, test, and deploy to GitHub Pages.

**New file** — `.github/workflows/deploy.yml`:
- Trigger: push to `main`
- Steps:
  1. Checkout
  2. Setup Node.js (LTS)
  3. `npm ci` (in `app/` directory)
  4. `npm run lint`
  5. `npm test`
  6. `npm run build`
  7. Deploy `app/build/` to GitHub Pages

**Custom domain:**
- CNAME file in `app/static/CNAME` with `rebbergverwaltung.haeusler-wein.ch`
- DNS: CNAME record pointing to `<org>.github.io`
- GitHub Pages settings: custom domain configured

**Base path:**
- With a custom domain at the root, no base path prefix is needed
- SvelteKit config: no `paths.base` override (defaults to `/`)

### 7. Scan results view (basic)

Show BBCH results after processing.

**Update** — scan detail page (`app/src/routes/vineyard/[id]/scan/[scanId]/+page.svelte`):
- After processing, show results table: Row | Vine | BBCH | Confidence
- Group by row number
- Color-code BBCH values (green for advanced, yellow for mid, etc.)

## Tests

### Unit tests (`tests/unit/ai/gemini.test.ts`)
- Parse valid Gemini response into BbchResult array
- Handle malformed response (missing fields, wrong types)
- Handle empty vine list (no vines detected)
- Validate vine_index is sequential starting from 1

### Integration tests (extend `tests/integration/storage/idb.test.ts`)
- BbchResult: save, load by scan, delete by scan
- VineMap: save entries, load by vineyard, delete by vineyard
- Cascade: deleting a scan also deletes its BbchResults

### Processor tests (`tests/unit/ai/processor.test.ts`)
- State machine transitions: LOCAL_ONLY -> UPLOADING -> PROCESSING -> DONE
- Error path: API failure sets FAILED status
- Results are stored on DONE
